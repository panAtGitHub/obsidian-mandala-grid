# 九宫格新架构蓝图

## 1. 这份文档是做什么的

这不是“随便聊聊重构想法”的笔记，而是一份可以真正拿来执行的 V2 架构蓝图。

目标只有一个：

- 让这个插件彻底从“历史树状列视图 + 后续叠加九宫格”的混合结构，收敛为“以九宫格为核心模型”的插件。

这份蓝图主要回答 6 个问题：

1. 这个插件以后到底以什么为核心数据模型。
2. `cell / views / ui / stores / helpers` 各自到底负责什么。
3. 3x3、9x9、nx9、7x9 之间怎样共享代码。
4. 哪些旧概念必须退出主舞台。
5. 哪些能力需要保留，但要换一套表达方式。
6. 迁移要怎么做，才能不中断现有插件可用性。

---

## 2. 当前问题总结

### 2.1 当前项目的根本问题

现在的代码已经不是单一架构，而是三层历史叠加：

1. 最早的树状列视图思路
2. 后来逐步长出来的 Mandala 九宫格视图
3. 为了兼容历史逻辑，不断在旧模型上打补丁

结果就是：

- 目录结构已经开始往职责分层走，但内部语义还没完全跟上
- 一部分 UI 已经是九宫格思维
- 一部分 store / helper / 命名还在按列视图思维工作
- 某些功能已经删掉 UI，但行为层仍然继承旧结构假设

### 2.2 目前最典型的问题

#### 问题 A：产品核心已是九宫格，但内部语义仍有旧列视图残留

比如：

- `column`
- `group`
- `activeBranch`
- 旧式对齐 / 滚动行为

这些概念不一定都错，但它们不应该再主导当前 UI 架构。

#### 问题 B：格子、视图、界面、状态边界还不够硬

虽然已经拆出了：

- `src/cell`
- `src/views`
- `src/ui`

但很多地方仍存在：

- 视图自己决定格子交互
- 格子组件知道视图模式
- helper 直接跨层取 DOM 或状态

#### 问题 C：功能是“逐步积累出来的”，不是按统一协议生长出来的

例如：

- 3x3、9x9、nx9、7x9 共性很多，但没有统一的视图协议
- 简化九宫格和标准卡片一度各自实现了一套格子系统
- 某些交互先写在组件里，后来再抽 helper

这会导致：

- 同样能力在多个地方实现
- 一处改动容易漏掉别处
- 后续任何新需求都必须先判断“要接哪套旧系统”

---

## 3. V2 架构目标

### 3.1 核心目标

新架构要做到这几点：

1. 插件核心语义只围绕“九宫格文档”和“格子视图”组织。
2. 单个格子的渲染、交互、状态输入必须独立。
3. 视图层只负责“布局”和“把文档映射成格子”。
4. 界面外围功能不能再反向控制格子内部实现。
5. 底层文档结构和 UI 展示语义明确分层。

### 3.2 非目标

这次新架构不追求：

- 一次性重写全部功能
- 先做大而全的抽象
- 在还没稳定前就全面优化性能

先要解决的是：

- 结构清晰
- 责任清晰
- 迁移可控
- 功能不断

---

## 4. V2 的核心设计原则

## 4.1 原则一：以“格子”为 UI 基本单位

以后 UI 层不再以：

- 列
- group
- branch

作为主要展示语义。

而是以：

- `cell`

作为最小展示与交互单位。

格子是这个插件的真正 UI 原子。

## 4.2 原则二：视图不操作格子内部，只装配格子

视图层只做：

- 一页有哪些格子
- 每个格子对应哪个 node / section
- 当前视图如何排布
- 给格子注入标准化输入和动作

视图层不做：

- 格子内部渲染细节
- 格子双击编辑细节
- swap 状态机细节
- 格子可滚动区域细节

## 4.3 原则三：文档模型和视图模型分离

要明确区分：

- 文档事实
- 视图态
- UI 临时态

不能继续让很多逻辑同时读写：

- document store
- view store
- plugin settings

而没有统一边界。

## 4.4 原则四：保留领域概念，但不让旧 UI 概念统治新架构

例如：

- `column/group` 作为底层结构仍然可以存在
- 但它们只属于“文档组织层”
- 不能再直接决定“当前界面怎么渲染”

---

## 5. V2 的分层模型

新架构建议分成 7 层。

## 5.1 第 1 层：Engine

路径建议：

```text
src/engine/mandala-document/
```

职责：

- 解析 Mandala 文档
- 构建结构化文档状态
- 序列化回 markdown/frontmatter
- 校验 section / slot / layout 合法性
- 提供纯领域层规则

这一层不应该知道：

- Svelte
- DOM
- 当前是哪种视图
- 当前哪个格子是 active

它只处理“文档是什么”。

## 5.2 第 2 层：Domain

路径建议：

```text
src/domain/mandala/
```

职责：

- day-plan 规则
- week-plan 规则
- section color 规则
- layout 规则
- swap 规则
- 预设 / profile / 模式兼容性规则

这一层比 engine 更接近产品语义，但仍然不应该碰 UI。

可以理解为：

- engine 解决“如何表示文档”
- domain 解决“文档业务规则是什么”

## 5.3 第 3 层：Stores

路径建议：

```text
src/stores/
  document/
  mandala/
  ui/
  settings/
```

职责：

- document：文档事实状态
- mandala：当前视图态，例如 active cell、subgrid、swap 状态
- ui：modal、toolbar、sidebar、preview dialog
- settings：用户偏好、布局、显示规则

这层是“状态容器”，不是“逻辑堆放区”。

要求：

- store 本身尽量薄
- 真正复杂的派生与决策交给 selector / controller

## 5.4 第 4 层：Cell

路径建议：

```text
src/cell/
  model/
  display/
  interaction/
```

职责：

- 单个格子的标准输入模型
- 单个格子的渲染块
- 单个格子的交互控制器

这是 UI 核心层。

### Cell 层应当包含什么

#### `model/`

例如：

- `card-view-model.ts`
- `card-render-model.ts`
- `simple-summary-cell-model.ts`

作用：

- 定义格子的标准输入
- 不参与 DOM
- 不参与 store 订阅

#### `display/`

例如：

- `mandala-card.svelte`
- `simple-summary-cell.svelte`
- `card-main-content.svelte`
- `card-meta.svelte`

作用：

- 纯展示
- 只吃 model / props
- 不自行决定复杂业务流程

#### `interaction/`

例如：

- `mandala-card-controller.ts`
- `swap-controller.ts`
- `cell-activation-policy.ts`

作用：

- 控制 click / dblclick / swap / edit / focus
- 与 DOM 事件衔接
- 但不负责排布视图

## 5.5 第 5 层：Views

路径建议：

```text
src/views/
  shared/
  view-3x3/
  view-9x9/
  view-nx9/
  view-7x9/
```

职责：

- 把文档映射成特定布局
- 决定当前一屏有哪些格子
- 构建格子的 view model
- 组合格子组件

### 视图层必须遵守的规则

视图层允许：

- 决定布局
- 决定 section -> node 映射
- 决定显示哪种格子组件

视图层不允许：

- 直接写格子内部渲染细节
- 直接写 swap 状态机
- 直接写复杂滚动容器逻辑
- 直接写多套格子交互

## 5.6 第 6 层：UI

路径建议：

```text
src/ui/
  toolbar/
  sidebar/
  modals/
  settings/
  shared/
```

职责：

- 工具栏
- 左侧栏
- 明细侧栏
- 各种弹层
- 设置页

UI 层不是业务核心，只是外围控制层。

它应当依赖：

- views
- cell
- stores

但不应该直接改写 cell 内部实现。

## 5.7 第 7 层：Helpers

路径建议：

```text
src/helpers/
  shared/
  views/
  dom/
```

只保留三类 helper：

1. 纯通用 helper
2. 视图层 helper
3. DOM/布局 helper

不再继续发展“到处都是 helper，但没人知道应该放哪”的状态。

## 7. 关键对象模型

## 7.1 文档事实模型

建议明确一个只服务文档事实的结构，例如：

```ts
type MandalaDocumentState = {
  nodes: Record<NodeId, MandalaNode>;
  sections: {
    bySection: Record<SectionId, NodeId | null>;
    byNode: Record<NodeId, SectionId>;
  };
  structure: {
    roots: SectionId[];
    layoutId: string | null;
  };
  metadata: {
    pinnedSections: string[];
    colorsBySection: Record<string, string[]>;
  };
};
```

这里不包含：

- active node
- active cell
- editing state
- modal open state

这些不属于文档事实。

## 7.2 视图态模型

建议将 Mandala 视图态统一收为：

```ts
type MandalaUiState = {
  mode: '3x3' | '9x9' | 'nx9' | 'week-7x9';
  subgridTheme: string | null;
  activeCell9x9: GridCell | null;
  activeCellNx9: Nx9Cell | null;
  activeCellWeek7x9: WeekCell | null;
  swap: SwapState;
  showDetailSidebar: boolean;
};
```

## 7.3 格子输入模型

所有格子都应通过标准化 view model 接入。

例如：

```ts
type MandalaCardViewModel = {
  nodeId: string;
  section: string;
  active: boolean;
  editing: boolean;
  selected: boolean;
  pinned: boolean;
  style: NodeStyle | undefined;
  sectionColor: string | null;
  preserveActiveBackground: boolean;
  sectionIndicatorVariant: 'plain' | 'plain-with-pin' | 'section-capsule';
  gridCell: CellGridPosition | null;
};
```

简化格子也应有自己的 model，而不是在视图里裸拼 class。

## 8. 四大视图的统一协议

## 8.1 所有视图都应实现同一套职责

每种视图都必须回答：

1. 本视图有哪些 cell
2. 每个 cell 对应哪个 section
3. 每个 cell 对应哪个 node
4. 用什么 cell renderer
5. 触发交互时应注入什么动作

## 8.2 建议统一成 view adapter

例如：

```ts
type ViewCellAdapter<TCell> = {
  getCells(): TCell[];
  getKey(cell: TCell): string;
  buildCardViewModel(cell: TCell): MandalaCardViewModel | null;
  buildSummaryViewModel?(cell: TCell): SimpleSummaryCellModel | null;
};
```

### 3x3

- 使用标准 `MandalaCard`
- 强调主题宫格与子宫格关系

### 9x9

- 标准模式：使用 `MandalaCard`
- 简化模式：使用 `SimpleSummaryCell`

### nx9

- 使用 `MandalaCard`
- 额外处理 page / future row / next core cell

### 7x9

- 桌面版使用 `MandalaCard`
- 移动版可保留轻量 cell，但也必须走 cell 层组件

## 9. 交互架构

## 9.1 交互不能再写散

以后所有格子相关交互只允许在两个位置出现：

1. `cell/interaction/controller`
2. `cell/interaction/policies`

组件模板里只能做：

- 事件绑定
- 传入上下文

不能再写长段业务判断。

## 9.2 建议拆成 4 类交互控制器

### A. Select Controller

职责：

- 激活 node
- 激活 cell
- 聚焦主容器

### B. Edit Controller

职责：

- 主区编辑
- 侧栏编辑
- preview dialog 抢占编辑器

### C. Swap Controller

职责：

- 进入 swap
- 目标合法性校验
- 执行交换
- 阻断双击

### D. Navigation Controller

职责：

- 进入子宫格
- 退出子宫格
- 并行九宫格切换
- nx9 page 导航

## 10. 滚动与聚焦架构

这是当前项目里最容易持续腐化的一层，所以必须单列。

## 10.1 当前问题

历史上很多逻辑默认：

- 当前节点在某一列里
- 要滚动某一列
- 要对齐父列 / 子列 / 同层列

这在九宫格模式下已经不成立。

## 10.2 新的统一思路

以后滚动/聚焦应只围绕两个对象：

1. 主容器
2. 当前格子的最近可滚动 pane

## 10.3 建议新增统一滚动策略层

例如：

```text
src/ui/scroll/
  focus-target.ts
  scroll-pane.ts
  fit-document.ts
  fit-focus.ts
```

职责：

- 找当前活跃格子的可滚动容器
- 计算当前文档整体可视区域
- 计算当前焦点集合的 bounding rect
- 提供统一滚动 API

不要再让：

- toolbar helper
- keyboard command
- align helper

各自发明一套滚动容器寻找逻辑。

---

## 11. 状态流设计

建议以后统一遵守：

### 文档事实

只允许通过：

- document reducer

改动。

### 视图态

只允许通过：

- mandala ui reducer

改动。

### UI 态

只允许通过：

- ui reducer

改动。

### 设置

只允许通过：

- settings reducer

改动。

不要继续出现：

- 一个 helper 顺手改 document
- 一个组件内部直接 dispatch 多个系统
- 一个 action 同时承担 UI + 业务 + 持久化

## 12. 命名规范

## 12.1 必须收敛的命名

### 不再推荐作为 UI 主语义的命名

- `column`
- `group`
- `branch`
- `container`

这些名字只允许留在底层结构层或迁移兼容层。

### 推荐的新命名

- `cell`
- `grid`
- `section`
- `subgrid`
- `view-model`
- `controller`
- `policy`
- `render-model`

## 12.2 文件命名建议

- `mandala-card.svelte`
- `simple-summary-cell.svelte`
- `build-mandala-card-view-model.ts`
- `cell-activation-policy.ts`
- `grid-focus-controller.ts`

尽量做到“看文件名就知道它属于哪一层”。

## 13. 需要退出主架构的旧概念

以下内容不一定全部立即删除，但必须退出核心路径：

1. 经典列视图 UI
2. 以 `.column` 为核心的滚动模型
3. 以 `ActiveStatus.node/parent/child/sibling` 为核心的显示语义
4. 以 `container.svelte` 为中心的旧视图壳层
5. “一个组件同时负责渲染、交互、模式判断、状态拼接”的大组件写法

## 14. 迁移策略

不建议一次性重写整个插件。

建议采用“分阶段内核替换”。

## 阶段 1：锁定旧系统边界

目标：

- 旧列视图彻底退出主路径
- 旧 `.column` 行为假设清理
- 给现有九宫格系统建立稳定边界

完成标准：

- 当前产品只剩一套真实可运行的九宫格 UI 主链

## 阶段 2：完成 Cell 层标准化

目标：

- 所有格子都走统一模型
- 所有格子交互都走 controller/policy
- 所有视图都不再自己写第二套 cell 系统

完成标准：

- cell 是真正独立层

## 阶段 3：完成 View Adapter 化

目标：

- 3x3 / 9x9 / nx9 / 7x9 都实现统一协议
- 视图只做布局与映射

完成标准：

- 视图层不直接操作格子内部

## 阶段 4：重做滚动与聚焦系统

目标：

- 将 `align-branch` 退役或重写
- 改为面向当前网格的 focus / fit / scroll controller

完成标准：

- 当前所有滚动/聚焦都不再依赖历史列视图语义

## 阶段 5：清理数据语义残留

目标：

- 收敛 `activeBranch / column / group` 等内部命名
- 让文档层和 UI 层的语义各归各位

完成标准：

- 新人看代码时，不会再误以为这个插件还是列视图插件

---

## 15. 第一批建议立即执行的任务

如果现在正式开始按新架构推进，我建议第一批任务只做这些：

1. 新建 `src/domain/mandala/`
2. 盘点 `src/lib/mandala/` 中哪些应迁到 domain
3. 新建 `src/ui/scroll/` 或等价滚动策略目录
4. 将 `fit document / fit active branch / scroll-node / align-branch` 收到统一焦点策略
5. 给四大视图补一层统一的 view adapter

不要第一轮就去：

- 大规模改 document reducer
- 一次性重写所有 store
- 一次性改所有命名

先把最容易腐化的“视图-格子-滚动”三角关系收稳。

### 15.1 当前执行批次：格子与视图层剥离

这批任务只做一件事：

- 把 section 编号、颜色指示、锁定指示，从“视图零散控制 + 内容隐藏信息残留”收回到 `cell`

执行顺序：

1. 给 `cell` 建立统一的显示策略 `displayPolicy`
2. 给 `cell` 建立统一的交互策略 `interactionPolicy`
3. `mandala-card` 只消费策略，不再让视图直接传 `sectionIndicatorVariant / preserveActiveBackground`
4. `content` 增加“隐藏内建 hidden-info”的能力，让 section 编号不再从内容区左下角漏出来
5. 迁改 `3x3 / 7x9 / nx9 / 9x9`
6. 清理视图侧旧 section 显示链
7. 后续再处理 `simple 9x9` 的特例格子系统

这批做完后的完成标准：

- 视图只传 preset / policy，不再传零散格子细节
- section 编号、颜色、锁定由 `cell` 自己决定怎么显示
- 内容区里的旧 section 编号显示链退出主路径

---

## 16. 最终目标图

最终应该形成这样的依赖方向：

```text
engine -> domain -> stores
stores -> cell
stores -> views
cell -> views
views -> ui
helpers 只做横向支持
```

更直白地说：

- engine 负责文档
- domain 负责规则
- stores 负责状态
- cell 负责格子
- views 负责布局
- ui 负责外围界面

如果某段代码说不清自己属于哪层，那通常说明它还没设计好。

---

## 17. 一句话结论

这个插件的新架构，不应该是“继续修补一个历史列视图插件”，而应该是：

**一个以 Mandala 九宫格文档为核心、以 cell 为原子、以 views 为布局层、以 ui 为外围控制层的插件。**
