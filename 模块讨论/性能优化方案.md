# Mandala Grid 性能优化方案

> [!summary]
> 本文基于 2026-03-23 对当前代码的静态体检整理，目标不是“零碎修补卡顿点”，而是形成一套可以分阶段落地、可验收、可回退的完整优化方案。

## 1. 当前结论

当前插件的卡顿，主因不是单一的大 bug，也不是磁盘写入本身，而是以下几类成本叠加：

- 一个小动作会触发过宽的状态失效范围，导致整屏重算。
- `3x3 / 9x9 / nx9 / week-7x9` 共享层承担了过多场景同步逻辑，不同模式互相牵连。
- `nx9`、`week-7x9` 桌面端会批量组装 `MandalaCard`，卡片本身又挂着 Markdown 预览、样式推导和交互状态。
- embed 刷新链路偏全局，打开多个 Markdown 预览页时容易放大卡顿。
- 保存与 settings 持久化链路虽然不是第一主因，但会在退出编辑、切文件和高频切换节点时补刀。

## 2. 优化目标

### 2.1 用户感知目标

- 常规文档中，切换活跃卡片不应有肉眼可见的整页抖动。
- 退出编辑并保存时，不应出现明显停顿。
- `nx9` 翻页、`9x9` 切核心九宫格、`week-7x9` 切活跃格时，应明显比当前顺滑。
- 工作区打开多个 Markdown 预览页时，修改一个九宫格文件，不应让无关预览一起重渲染。

### 2.2 技术目标

- 将主线程热点从“整视图重算”改成“最小必要范围更新”。
- 将 embed 刷新从“扫描所有打开视图”改成“命中受影响视图”。
- 将保存链路从“尽量 patch，失败再全量重建”进一步推进为“单 section 变更尽量只按索引 patch”。
- 将高频 UI 状态从“即时写 settings”改成“运行时缓存 + 合适时机落盘”。

## 3. 优化原则

### 3.1 先收窄失效范围，再谈算法优化

如果一次交互仍然会让整页所有组件都进入更新路径，那么即使单个函数再快，整体体感也不会好。

### 3.2 先做低风险高收益改造

优先调整 selector、场景同步边界、卡片级 DOM 读取和 embed 刷新范围；不要一上来全仓改 immutable store。

### 3.3 每一轮都要可测量、可回退

每个阶段都要有性能基线、触发路径、验收口径和回退点，避免“做了很多优化但体感不确定”。

## 4. 关键问题分层

## 4.1 状态失效范围过大

当前多个布局直接订阅整个 `documentStore` 或整个 `viewStore`，导致很多和当前模式无关的 action 也会引发重算。

典型表现：

- 场景共享层同时处理 `subgridTheme`、`activeCell`、`weekAnchorDate`、布局、移动端编辑视口等状态。
- `9x9`、`nx9`、`week-7x9` 的视图组装普遍依赖整块状态对象，而不是精确字段。

直接后果：

- 任意一个局部状态变化，都可能让整页 reactive 链路重新走一遍。

## 4.2 共享层承担了过多模式同步

当前共享视图层不是单纯壳层，而是“总控 + 模式同步器 + UI 协调器”的混合体。

直接后果：

- `3x3` 的逻辑变化可能牵出 `nx9` 或 `week-7x9` 的状态检查。
- 同一份 reactive 输入会同时喂给多个模式的同步逻辑。

## 4.3 卡片渲染成本偏高

`MandalaCard` 不只是一个轻量展示组件，它还承担：

- render model 组装
- 主题色与背景样式推导
- 预览弹窗联动
- 交互状态判断
- Markdown 内容渲染入口

直接后果：

- 在 `nx9` 和 `week-7x9` 这种多卡场景中，任何父层重建都会被放大。

## 4.4 Embed 刷新链路过宽

embed 刷新当前更像“全局观察 + 事后判定”，而不是“精确命中受影响视图”。

直接后果：

- 工作区 Markdown 预览页越多，刷新链路越重。
- `forceAll` 分支代价偏大，容易造成工作区级抖动。

## 4.5 保存和持久化还在补刀

虽然主编辑内容不是每敲一个字就全量保存，但退出编辑后仍可能触发：

- section patch 扫描
- `prepareSaveSections`
- `serializeSections`
- snapshot 持久化
- settings 保存排队

直接后果：

- 在大文档或频繁切换节点时，保存动作会被感知为“顿一下”。

## 5. 完整优化路线

## 5.1 P0：建立性能基线与预算

### 目标

先让后续优化具备统一的评估口径。

### 动作

- 补齐现有 perf 日志的使用场景，重点覆盖：
  - 切文件
  - 首次渲染
  - 切活跃节点
  - 退出编辑保存
  - `9x9` 切核心九宫格
  - `nx9` 翻页
  - embed 刷新
- 为每个场景建立固定测试样本：
  - 小文档
  - 中等文档
  - 大文档
  - 多预览页 + 多 embed
- 记录改造前数据，作为每一轮回归对比基准。

### 预算建议

- 常规交互主线程耗时：`< 16ms`
- 退出编辑保存：`< 50ms`
- 切文件首屏：`< 150ms`
- 单次 embed 刷新：只影响命中的视图

### 预期收益

- 后续每轮优化都能判断是否真正有效。

## 5.2 P1：缩小状态失效范围

### 目标

把“一个动作触发整页重算”改成“只有相关字段变化才更新”。

### 动作

- 为热点视图建立更细粒度的 selector，不再直接订阅整个 `documentStore` / `viewStore`。
- 为 selector 增加显式 equality，避免仅因对象引用不稳定而重算。
- 优先拆分以下几类输入：
  - 活跃节点
  - 当前模式
  - 当前场景所需 section map
  - 当前场景所需 active cell
  - 当前场景所需布局配置
  - 颜色映射与 pinned 状态
- 只在真正需要的场景中订阅对应字段，不在共享层统一拿整块状态。

### 重点模块

- `src/mandala-scenes/shared/mandala-view.svelte`
- `src/mandala-scenes/view-9x9/layout.svelte`
- `src/mandala-scenes/view-nx9/layout.svelte`
- `src/mandala-scenes/view-nx9-week-7x9/layout.svelte`

### 预期收益

- 主线程更新链显著变短。
- 很多“明明只是切换一个格子，为什么整个视图都在动”的情况会直接消失。

### 风险

- 如果 selector 切得过细，容易漏掉某些边界状态。

### 回退策略

- 每个场景单独改造，按视图逐步切换，不做全仓一次性替换。

## 5.3 P2：拆共享层的场景同步

### 目标

把共享层从“总控逻辑聚集地”收缩成真正的壳层。

### 动作

- 把 `3x3`、`9x9`、`nx9`、`week-7x9` 的场景同步逻辑拆回各自目录。
- 将当前共享层中以下逻辑按模式拆分：
  - active cell 同步
  - 模式兼容性检查
  - subgrid/theme 修正
  - week anchor 同步
  - `nx9` 上下文同步
- 对所有会触发 dispatch 的同步逻辑，增加“输入未变则不 dispatch”的保护。

### 重点模块

- `src/mandala-scenes/shared/mandala-view.svelte`
- `src/mandala-scenes/shared/sync-scene-state.ts`
- `src/mandala-scenes/view-3x3/scene-state.ts`

### 预期收益

- 不同模式之间的状态联动被切断。
- 场景逻辑边界更清晰，后续维护成本降低。

## 5.4 P3：瘦身各视图的 view model 组装

### 目标

避免 `3x3 / 9x9 / nx9 / week-7x9` 在普通交互中整表重建。

### 3x3

- 保持现有结构，不追求大改。
- 给 `assemble3x3CellViewModels` 加输入稳定化与缓存。
- 重点避免因背景色、选中状态、活跃状态等变化导致 9 个格子全量重建。

### 9x9

- 让 81 格重建只发生在以下场景：
  - 当前 core 变化
  - 布局切换
  - 颜色配置变化
  - 相关 section 内容变化
- 将“内容摘要提取”和“颜色装饰”拆成两段缓存，不要每次同时重做。
- 把主题色读取从 cell 装饰阶段移到更上层的 theme store。

### nx9

- 缓存 `resolveNx9Context()` 的结果。
- active cell 变化时，只更新当前页相关行，不重算整页 view model。
- 翻页时只组装目标页，不重复推导无关页面。

### week-7x9

- 桌面端重点减少 `assembleDesktopWeekPlanCells` 全表重组。
- 移动端保持摘要策略，但也要避免全表重复提取 title/body。

### 预期收益

- `9x9` 和 `nx9` 的体感提升最明显。
- `week-7x9` 桌面端的“切格子就整排动一下”问题会明显缓解。

## 5.5 P4：降低卡片本身的渲染成本

### 目标

让 `MandalaCard` 从“重型综合组件”变成“尽量吃稳定输入的展示层”。

### 动作

- 把 `themeTone`、背景 underlay 色等全局信息提升到 view 级 store，不在每个卡片里读 DOM。
- 将 render model 拆成：
  - 稳定部分：section、displayPolicy、interactionPolicy、基础样式
  - 高频部分：active、selected、editing、swap 状态
- 保持卡片组件实例稳定，避免父层 view model 重新 new 对象时导致内容层无谓重挂载。
- 对卡片内部的 Markdown 预览入口加“内容未变不重渲”的保护。

### 重点模块

- `src/mandala-cell/view/components/mandala-card.svelte`
- `src/view/actions/markdown-preview/markdown-preview-action.ts`
- `src/view/actions/markdown-preview/helpers/render-markdown-content.ts`

### 预期收益

- 多卡场景下的累计收益很大。
- 对 `nx9`、`week-7x9`、detail sidebar 都有正向影响。

## 5.6 P5：重构 embed 刷新链路

### 目标

让 embed 刷新从“全局扫描”变成“精确命中”。

### 动作

- 建立 source -> target、target -> source 的运行时索引。
- 打开 MarkdownView 时记录其包含的 mandala embed 依赖关系。
- 文件变化时直接命中受影响的 preview / live preview，而不是扫描所有打开的 MarkdownView。
- 收窄 `forceAll` 触发条件，能定向刷新时绝不走全局刷新。
- 评估 `workspace.updateOptions()` 的必要性，避免把局部刷新升级成工作区级刷新。

### 重点模块

- `src/obsidian/events/workspace/register-mandala-embed-refresh-events.ts`
- `src/main.ts`
- `src/obsidian/markdown-post-processors/mandala-embed/**`

### 预期收益

- 工作区里开多个 Markdown 预览页时，卡顿会明显下降。
- embed 越多，收益越明显。

## 5.7 P6：优化保存与持久化链路

### 目标

降低退出编辑、切文件和局部修改时的字符串扫描与写盘成本。

### 动作

- 加载文档时额外建立 section range index。
- 单 section 内容变更时，优先按 offset patch，不再每次重新按行扫描 marker。
- 只有结构变化时才进入 `prepareSaveSections + serializeSections`。
- 将高频 UI 状态改为运行时缓存，优先在以下时机落盘：
  - 退出编辑
  - 切文件
  - 关闭视图
  - 闲时
- 降低 `activeSection` 等高频 settings 写入频率。

### 重点模块

- `src/view/view.ts`
- `src/mandala-display/logic/apply-section-patch.ts`
- `src/mandala-document/engine/prepare-save-sections.ts`
- `src/main.ts`

### 预期收益

- 大文档退出编辑时的停顿会下降。
- 高频节点切换不再不断把 settings 持久化队列堆起来。

## 5.8 P7：大文档专项与可选 worker 化

### 目标

解决极端文档规模下的边界性能问题。

### 动作

- 评估整文档 parse/build 是否需要移到 worker。
- 对超大文档场景建立专门压测样本。
- 只在前面几轮完成后，再决定是否引入 worker 化解析。

### 原因

- 这一步收益取决于前面主线程无谓重算是否已经被压下去。
- 若前面没做好，worker 化只能缓解局部，并不能解决整体卡顿。

## 6. 实施顺序建议

> [!important]
> 建议严格按顺序推进，不要跳着做。

### 第一轮

- 建立性能基线
- 缩小状态失效范围
- 拆共享层同步

### 第二轮

- 瘦身 `9x9`
- 瘦身 `nx9`
- 瘦身 `week-7x9`
- 卡片级 DOM 读取上提

### 第三轮

- Markdown 预览链去重
- embed 刷新重构
- settings 持久化降频

### 第四轮

- 保存链路进一步索引化
- 大文档专项
- 评估是否引入 worker 化解析

## 7. 不建议优先做的事

- 不建议一开始就把全仓 store/reducer 重写成 immutable。
- 不建议先从 CSS 微调入手。
- 不建议先上 worker 化 embed 渲染。
- 不建议把所有性能问题都归因到磁盘保存。

## 8. 验收标准

### 功能不回退

- `3x3 / 9x9 / nx9 / week-7x9` 四种模式行为一致。
- active node、active cell、subgrid、week anchor、sidebar、preview dialog 不出现同步错误。
- embed 打开、刷新、跳转、复选框交互都正常。

### 性能达到目标

- 常规文档下切换节点无明显整页重绘感。
- 退出编辑后的保存耗时明显下降。
- `nx9` 翻页、`9x9` 切核心、`week-7x9` 切格子都优于当前版本。
- 多 Markdown 预览页下，只刷新受影响视图。

### 工程可维护

- 场景逻辑尽量回到 `view-xxx/` 目录。
- 共享层只保留真正跨场景的壳层逻辑。
- 性能优化点有注释、有日志口径、有测试样本。

## 9. 最终建议

当前最值得先动的，不是“再抠某一个算法函数”，而是：

1. 缩小 store 订阅范围。
2. 拆共享层同步。
3. 降低 `MandalaCard` 和 Markdown 预览的重渲染概率。
4. 把 embed 刷新从全局扫描改成精确命中。

如果这四步做对，体感上的卡顿大概率会先消掉一大半；后续再做保存链路和大文档专项，收益才会真正稳定。
